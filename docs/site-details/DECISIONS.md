# Site Details Decisions

## Canonical service-location decision
- Decision: evolve current location lineage into canonical `ServiceLocation` semantics (single model lineage).
- Rationale: avoid dual live location systems, preserve existing data, support pre-customer and post-customer lifecycle continuity.

## Address snapshot vs operational truth
- `Lead.address` remains intake snapshot/audit input.
- Operational address truth after linkage is canonical service location record.
- Quote/job no longer rely on copied address text in notes as operational truth.

## Phase 1 structure (locked)
- Phase 1A: additive schema only, no legacy removal.
- Phase 1B: backfill + reconciliation with explicit ambiguity handling.
- Phase 1C: canonical read/write cutover.
- Phase 1D: legacy freeze/removal after parity checks.

## Migration safety (locked)
- Preserve existing table data.
- No migration that drops/recreates location table during rename/evolution.
- Review generated SQL and risk before apply.
- Stop on destructive SQL or unresolved mapping risk.

## Location stability rules (locked)
- `Quote.serviceLocationId` is explicit and stable.
- `Job.serviceLocationId` is explicitly copied at quote activation.
- Customer primary-location changes do not mutate existing quote/job links.
- Quote/job location reassignment must be explicit, authorized, and audited.
- Material address changes set stale state; facts do not silently migrate across properties.

## ZIP utility semantics
- ZIP coverage is likelihood evidence, not property confirmation.
- Multiple utility candidates per ZIP are supported.
- Utility-wide links belong to Utility, not coverage rows.

## Link ownership
- Utility coverage evidence source belongs to UtilityCoverage.
- Utility official process links belong to Utility.
- Jurisdiction/department links belong to Jurisdiction/Authority.
- County APN lookup belongs to CountyAssessorResource.
- Exact APN belongs to canonical service location.

## User correction priority
- Priority hierarchy:
  - `USER_CORRECTED`
  - `USER_REVIEWED`
  - `AI_FOUND`
  - `DATABASE_MATCH`
  - `UNVERIFIED`
- AI and later refreshes never silently overwrite reviewed/corrected values.

## Organization isolation
- MVP knowledge writes are organization-scoped.
- All reads/writes enforce `organizationId`.
- No cross-organization shared writes in MVP.
- Site details are not exposed in public quote-sharing routes.

## Grounded-search boundary
- Reuse existing Gemini AI boundary in `ai-service`.
- Add dedicated `researchSiteDetails(...)`.
- Grounded search only for missing reusable scopes.
- Do not blend search behavior into normal execution-plan generation.

## Manual APN decision
- APN is not inferred/generated by AI in MVP.
- User enters APN manually after official assessor lookup.
- APN writes are append-only audited.

## Execution integration rule
- Site details feed execution context as status-qualified facts.
- Known details should suppress redundant generic utility/jurisdiction unknowns.
- Scope/process specifics remain unresolved unless explicitly verified.

## Audit coverage (append-only)
- Track old/new/source-reason for:
  - APN changes
  - utility assignment changes
  - jurisdiction assignment changes
  - address updates
  - quote/job location reassignment
  - review status changes
  - AI-found acceptance/rejection/replacement
